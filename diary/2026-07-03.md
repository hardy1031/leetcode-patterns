## Theme: HashMap + Doubly Linked List for O(1) ordered access

Started from the question "why do we need left and right pointers?" in LRU Cache. The real question turned out to be: how do you maintain recency order while keeping both lookup and reordering at O(1)?

### Key insight
HashMap gives O(1) lookup; doubly linked list gives O(1) remove+insert anywhere — combining them is the only way to satisfy both constraints simultaneously. Singly linked list fails because `remove(node)` requires `node.prev`, which you can't get in O(1) without backward pointers.

## Theme: Choosing data structures for O(1) guarantees

LRU Cache forces the question: "how do you maintain order while keeping reordering at O(1)?" The answer is DLL+HashMap — but the broader skill is knowing which structure to reach for based on what you need to do in O(1).

### Key insight
Every O(1) requirement maps to a specific structure. When a problem demands O(1) for some operation, use this as a lookup table:

| O(1) requirement | Structure |
|---|---|
| lookup by key | HashMap |
| min or max | Heap |
| ordered access + reorder | Doubly Linked List |
| push / pop (LIFO or FIFO) | Stack / Queue |

When a problem requires two O(1) operations simultaneously (e.g. lookup AND reorder), combine the corresponding structures.

## Theme: Sentinel nodes in doubly linked lists

Encountered `left` and `right` sentinel (dummy) nodes for the first time. The concept is identical to the dummy head used in singly linked list problems — a boundary node that holds no real data but eliminates edge cases.

### Key insight
With sentinels, `right.prev` and `left.nxt` always point to a valid node, so `insert` and `remove` never need special-case logic for empty lists or boundary positions. Same motivation as dummy node in singly linked list, extended to both ends.

## Theme: Tree DFS sub-patterns — basic vs. path_problems

Realized that tree DFS problems split into two distinct sub-patterns based on what the recursive function returns.

### Key insight
- **basic** (invert, max depth): the recursive return value IS the answer directly.
- **path_problems** (diameter): the recursive return value is a helper metric (height). The real answer is updated as a side effect via `nonlocal`. This separation is necessary when the answer spans left and right subtrees simultaneously.

## Theme: BFS iterative — deque takes an iterable, not a single element

Learned that `deque(root)` fails because `TreeNode` is not iterable. The correct form is `deque([root])` — wrapping the single element in a list first.

### Key insight
`deque(x)` iterates over `x`. To initialize with one element, pass `[x]` (a list) or `(x,)` (a tuple).

## Theme: DFS iterative — carry state as tuples in the stack

For max depth, BFS requires batching nodes by level (`for _ in range(len(queue))`). DFS avoids this by pushing `(node, depth)` tuples onto the stack — depth travels with the node, no level-batching needed.

### Key insight
When you need to track a value per node (depth, path sum, etc.) in iterative DFS, store it as a tuple alongside the node in the stack.

## Theme: nonlocal keyword in Python

First encounter with `nonlocal`. Without it, assigning to a variable inside a nested function makes Python treat it as a new local variable, causing an `UnboundLocalError` when you read it before assignment.

### Key insight
Use `nonlocal x` inside a nested function whenever you need to assign (not just read) a variable defined in the enclosing scope.
